# Seed Masking — How Deterministic Anonymization Works

This guide explains the masking seed in depth: what it does, how consistency
across files and days is guaranteed, what can silently break it, and how to
manage seeds as a team. Written for both technical and non-technical readers.

---

## 1. The one rule that explains everything

Every masked value is computed fresh, at the moment of masking, as:

```
masked_value = f(seed, rule, input_value)
```

Nothing else goes in. Not the file name, not the field name, not the field's
position in the record, not the date, not the record number, not any database
or stored mapping. There is **no lookup table anywhere** — the same three
inputs always recreate the same output, and changing any one of the three
changes the output completely.

Internally, `f` is an HMAC-SHA256 keyed by the seed; its output seeds a
random generator that drives the chosen rule. You never need to know that —
but it is why the guarantees below hold mathematically, not by convention.

## 2. What this guarantees

| Scenario | Result |
|---|---|
| Same value masked twice in one run | Same masked value |
| Same value in two different files (customer + mortgage) | Same masked value |
| Same value in fields with different names (`CUST-ID` vs `XYZ`) | Same masked value |
| Same value masked on day 1 and day 400 | Same masked value |
| Same value, files processed in any order | Same masked value |
| Regenerating a lost output file with the same seed | Byte-identical file |
| Same value, **different seed** | Completely different masked value |

**Join keys survive.** If customer, account, and mortgage files are all
masked with the same seed and the same rule on their customer-ID fields, a
given customer's ID maps to one masked ID everywhere, and every join your
test environment performs on real data still works on masked data.

**Staggered deliveries work.** If a mortgage file on day 1 contains customer
IDs whose customer records only arrive on day 2, mask both with the same
seed: the day-2 customer records will line up with the day-1 mortgage
records retroactively. Order of arrival is irrelevant.

## 3. The three things that CAN break consistency

Determinism only holds when all three inputs match. In practice these are
the failure modes to guard against:

### 3.1 Different rule
Same seed + same value + **different rule** = unrelated outputs.
"Replace digits" and "Scramble letters/digits" applied to the same ID give
two different masked IDs. Before generating, check the step-3 grid and make
sure every field that must join uses the **same rule in every file**
(recommendation: customer identifiers always use *Replace digits*).

The wizard suggests rules from the field's *name* (SIN, NAME, CARD, DOB, …).
An unrecognized name like `XYZ` falls back to a type-based default (numeric →
Replace digits, text → Scramble). Suggestions are a convenience — the grid is
the contract. Review it.

### 3.2 Different decoded value
The rule sees the value **as decoded from the file**, and the whole string
matters. Watch for:

- **Type/width drift between copybooks.** `PIC 9(08)` decodes `20000055` but
  `PIC X(10)` decodes `20000055  ` (trailing spaces). Different strings →
  different masked outputs. Keep shared IDs declared with the same type and
  width across copybooks, or at least be aware of the drift.
- **Leading zeros.** Numeric display fields decode zero-padded to their full
  width (`00012345` for a `9(08)`), consistently on both sides — so numeric ↔
  numeric is safe; numeric ↔ text of a different width may not be.

### 3.3 Synthesized records
When you request more output records than the input contains, the extras are
generated by re-masking input records with a per-cycle salt — deliberately
producing *different* values so synthetic records don't duplicate real ones.
Consequence: **synthetic records do not join across files.** For cross-file
consistency testing, keep the output count at or below the input count; use
synthesis when you need volume, not linkage.

## 4. Managing seeds as a team

- **One seed per test-data universe.** Typically one per environment or per
  release cycle. Every file that must interoperate is masked with that seed.
- **Store it safely**, named clearly (e.g. `anonymizer-seed-SIT-2026Q3`).
  Never email it alongside masked data, never commit it to a repo, never put
  it in a file share next to the outputs.

### The built-in team seed vault

The wizard's step 4 has a **"🔐 Team seed vault"** panel that solves the
"where do we keep the seed?" problem without any extra infrastructure:

- The vault is a **single encrypted file** (`seedstore.enc`). Put it on a
  shared network drive so the whole team uses one vault. Point the app at it
  either in the panel's "Vault file location" field or via the
  `ANONYMIZER_SEEDSTORE` environment variable; the default is
  `~/.mainframe_anonymizer/seedstore.enc` (personal, single-user).
- It is protected by a **team passphrase**: the passphrase is stretched with
  PBKDF2-HMAC-SHA256 (600,000 iterations) into the encryption key. Without
  the passphrase the file is unreadable, so it is safe to store on a shared
  drive. Saving to an existing vault also requires the correct passphrase.
- **Workflow:** the first person generates a seed, opens the vault panel,
  enters the team passphrase, and saves the seed under a name like
  `SIT-2026Q3`. Everyone masking a related file afterwards opens the same
  vault, picks `SIT-2026Q3` from the dropdown, and clicks *Use this seed* —
  no copy-pasting seeds over chat or email.
- **Honest limit:** the vault is as strong as the passphrase. Use a long
  one, share it only through your normal secure channel, and if your
  organization has an enterprise secret manager (CyberArk, Vault, Azure Key
  Vault), that remains the gold standard — this file is the pragmatic
  team-level alternative.
- **The audit report contains only a fingerprint** (first 8 hex chars of
  SHA-256 of the seed) — enough to answer "were these two files masked with
  the same seed?" without exposing the seed. Compare fingerprints across
  audit reports when in doubt.
- **Losing the seed** loses nothing already generated, but you can no longer
  produce *new* files consistent with the old ones, nor regenerate them.
  You'd start a fresh universe with a new seed.
- **Rotating the seed** (e.g., if it leaked) means re-masking every file in
  the universe with the new seed; old and new outputs will not join.

## 5. Security properties (and honest limits)

- Without the seed, masked values cannot be reversed to the originals: the
  mapping is an HMAC, and the output digits carry no recoverable information
  about the input.
- **With** the seed, anyone can replay the masking of a *guessed* input and
  check whether it matches a masked value (a "dictionary" check). This is
  exactly why the seed is treated like a password.
- Masking is not encryption and there is no decryption. If you need to
  recover originals, you need the original files — by design.
- Fields left as "Keep unchanged" pass through byte-for-byte. The tool masks
  what you tell it to mask; the step-3 review is the human checkpoint.

## 6. Fine print worth knowing

- **Masked values are not guaranteed unique.** Two *different* input IDs can
  in principle mask to the same output (like any random 8-digit assignment,
  collisions get likelier as volumes grow — the birthday effect). Real IDs
  map 1-to-1 consistently, but if a downstream system enforces uniqueness on
  a masked column across millions of records, be aware collisions are
  possible.
- **A masked value never equals its input.** Every rule guarantees the
  output differs from the input, including edge cases (values at the field's
  maximum, sentinel dates like `99991231`).
- **Checksums are preserved where they matter.** Masked SINs and card
  numbers are Luhn-valid, so downstream validators accept them.
- **Format is preserved.** Outputs fit the field's exact width, digit
  capacity, and encoding; dates stay parseable dates; signs stay valid.
- **The seed never appears in any output** — not in the masked file, not in
  the audit report (fingerprint only), not in logs.

## 7. Quick FAQ

**Q: New file, same customer_id but the field is called `XYZ`?**
Field names are irrelevant to the math. Just confirm the rule on `XYZ`
matches the rule used elsewhere (Replace digits) and use the same seed.

**Q: File arrives days/weeks later — will IDs still match?**
Yes. Determinism is time-independent. Same seed + rule + value = same
output, whenever it runs.

**Q: Can I mask the same file twice and diff the outputs?**
Yes — byte-identical with the same seed and settings. That's a useful
integrity check.

**Q: Do EBCDIC and ASCII versions of the same data mask the same way?**
Yes. Masking operates on decoded values, not raw bytes, so `JOHN SMITH` in
cp037 and in ASCII masks to the same name (then re-encodes per file).

**Q: What seed should I type?**
Any string works. Longer random strings (the pre-filled 16-hex value is
fine) resist guessing better than words. What matters most is reusing the
*same* one across related files and storing it safely.
